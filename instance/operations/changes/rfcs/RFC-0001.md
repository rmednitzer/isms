# Asset awareness, physical security, and data classification additions

**RFC-0001 Revision 1 (DRAFT)**

*Status: draft. Owner: role:ISMS-Manager. Next review: 2027-04-19.*

## 1. Context

ET-CORE-001 currently attests A.5.9 against repository markdown files. This is
a circular reference: the inventory-of-information-assets control evidences
itself against governance documents rather than against the actual information
assets, information systems, facilities, networks, suppliers, or personal data.
Several other applicable Annex A controls (A.5.12, A.5.13, A.5.19 through
A.5.23, A.7.1 through A.7.14, A.8.20 through A.8.22) have implementation
stubs with TODOs and no structured substrate. GDPR Article 30 records of
processing activities have no skeleton at all.

## 2. Proposal

Add six schema-backed registers, one governance standard, and supporting
tooling:

1. `template/governance/data/STD-005-data-classification-standard.md`: levels
   (public, internal, confidential, restricted), handling rules per level,
   labelling conventions.
2. `template/governance/assets/register.yaml` with
   `tooling/schemas/asset-register.schema.json`: information, software,
   hardware, service, facility, personnel, supply asset classes.
3. `template/governance/assets/facilities.yaml` with
   `tooling/schemas/facilities-register.schema.json`: sites, rooms, zones,
   physical controls by zone class.
4. `template/governance/assets/networks.yaml` with
   `tooling/schemas/network-register.schema.json`: segments, trust levels,
   encryption baseline, peering, ingress/egress policy posture.
5. `template/governance/supply-chain/register.yaml` with
   `tooling/schemas/supplier-register.schema.json`; migrate the inline
   `supply_chain.critical_suppliers` example out of `instance/config.yaml`.
6. `template/governance/data/inventory.yaml` with
   `tooling/schemas/data-inventory.schema.json`: GDPR Article 30 fields
   plus cross-references to asset and supplier registers.

Supporting tooling:

- `tooling/validators/validate_registers.py`: schema conformance plus
  cross-register reference resolution.
- Extend `tooling/validators/validate_crossrefs.py` to resolve register
  references from governance artefacts.
- Replace `tooling/collectors/core/inventory_from_repo.py` with
  `tooling/collectors/core/inventory_from_register.py`, which attests A.5.9
  against the asset register.
- Evidence plan: add ET-CORE-002 through ET-CORE-006 binding the new
  registers to review cadences.

## 3. Alternatives considered

- **Do nothing, close with a risk acceptance.** Rejected. The ET-CORE-001
  circular reference would appear as an audit finding at stage 2 and any
  supervisory review under NISG 2026 section 33.
- **Inline the content in existing policies without schemas.** Rejected.
  No validator coverage, no cross-reference integrity, no machine-readable
  path to the Selbstdeklaration package.
- **Import from an external CMDB.** Rejected. Breaks the standalone-operation
  charter (DEC-2026-001). External CMDB may be referenced from register
  entries but the register itself is the governance source of truth.

## 4. Impact (CACE)

- **Blast radius.** Template layer. All six files are additive under new paths.
  `instance/config.yaml` loses `supply_chain.critical_suppliers` inline example
  and gains a pointer. Existing instances, if any have been cut, re-render
  idempotently and pick up the new template paths as empty templates ready to
  populate.
- **Dependencies.** Risk register `asset_refs` field gains a resolution target.
  SoA implementation statements for A.5.9, A.5.12, A.5.13, A.5.19 through
  A.5.23, A.7.1, A.7.2, A.7.4, A.8.20, A.8.22 should be updated in a follow-up
  revision to reference the registers (not in this RFC).
- **Failure modes.** Schema evolution: if a register schema needs to change,
  follow the usual contribution flow with a revision bump. Name conflicts:
  ID prefixes are new; no conflict with existing DOC/P/SOP/STD/PLAN/REC/INC/
  IA/DEC/RFC/TMPL spaces.
- **Rollback.** `git revert` of the merge commit removes all additions
  cleanly. Re-rendering instances yields the pre-RFC state.
- **Stop conditions.** Halt if `make validate` or `make test` regress during
  any phase. Halt if the crossref validator detects unresolved references in
  the template sample entries.
